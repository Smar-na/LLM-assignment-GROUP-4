# Toxic Comment Classification using LLM Prompt Engineering

Multi-label toxic comment classification on the [Jigsaw Toxic Comment Classification dataset](https://www.kaggle.com/competitions/jigsaw-toxic-comment-classification-challenge/data) using GPT-4.1 prompt engineering pipelines and BERT-based baselines. Six toxicity labels are classified: `toxic`, `severe_toxic`, `obscene`, `threat`, `insult`, `identity_hate`.

## Data

### Download the Jigsaw dataset

Download `train.csv` from Kaggle:

```bash
kaggle competitions download -c jigsaw-toxic-comment-classification-challenge
unzip jigsaw-toxic-comment-classification-challenge.zip
```

Or download manually from:
https://www.kaggle.com/competitions/jigsaw-toxic-comment-classification-challenge/data

Place `train.csv` in the `data/` directory.

### Generate the evaluation and few-shot pool files

Run the stress-test sampling script to produce the evaluation set and few-shot pool:

```bash
python stress_test_sampling.py
```

This script:
- Reserves 20 examples per rare class (`threat`, `identity_hate`, `severe_toxic`) for the few-shot pool — strictly excluded from the evaluation set (zero leakage)
- Produces a stress-test evaluation set with all available rare-class comments plus matched clean comments (~8,000 rows, 50/50 toxic/clean split) — the definitive benchmark used consistently across all models
- Excludes 3 manually specified comment IDs from the RAG pool
- Applies text cleaning: drops null comments, handles encoding errors, and strips control characters
- Saves `stress_test_eval_set.csv` and `few_shot_pool_fixed_3.csv`
- Generates EDA visualisations of class distribution

> **Note:** The BERT notebook uses the same 8,000-row `stress_test_eval_set.csv` as all other models, ensuring a consistent apples-to-apples comparison across BERT and GPT-4.1 pipelines.

## Running the Pipelines

All notebooks are self-contained and designed for Google Colab or a local Jupyter environment. Each notebook includes checkpoint/resume support — if a run is interrupted, re-running it will automatically skip completed rows and continue from where it left off.

---

### BERT Baselines — `BERT.ipynb`

**Required files:** `stress_test_eval_set.csv`

Run all cells in order. The notebook will:
1. Load the 8,000-row stress-test evaluation set
2. Run inference with `google-bert/bert-base-cased` (randomly initialised 6-label head, `num_labels=6`, threshold=0.5) — uninstructed lower-bound baseline
3. Run inference with `unitary/toxic-bert` (domain-adapted model fine-tuned on Jigsaw, threshold=0.5) — supervised upper-bound baseline
4. Output per-label Precision, Recall, F1, and Accuracy reports for both models on the stress-test benchmark

> **Note:** `bert-base-cased` was chosen over `bert-base-uncased` because case sensitivity matters for toxicity detection — users frequently capitalise in toxic comments. `unitary/toxic-bert` is the state-of-the-art BERT model fine-tuned directly on the Jigsaw dataset.

---

### Method 1 — Zero-Shot — `Finalmethod1_zero_shot.ipynb`

**Required files:** `stress_test_eval_set.csv`

**Output:** `predictions_zero_shot.csv`

Run all cells in order. The notebook will:
1. Load the evaluation set
2. Run async GPT-4.1 inference (`ASYNC_CONCURRENCY=8`, `temperature=0`, `seed=42`)
3. Apply per-label confidence thresholds on a 0–100 scale to produce binary predictions
4. Save predictions with confidence scores, token counts, and per-row runtime
5. Print a full evaluation report: per-label F1, ROC-AUC, Hamming Loss, and cost breakdown

---

### Method 2 — Few-Shot Simple — `Finalmethod2_few_shot_simple.ipynb`

**Required files:** `stress_test_eval_set.csv`

**Output:** `predictions_few_shot41.csv`

Run all cells in order. The notebook will:
1. Load the evaluation set
2. Inject 41 static few-shot examples covering all observed label combinations in the Jigsaw dataset into every prompt
3. Run async GPT-4.1 inference
4. Save predictions and print full evaluation report

---

### Method 3 — RAG Dynamic — `FINAL_METHOD3_RAGDYNAMIC.ipynb`

**Required files:** `stress_test_eval_set.csv`, `few_shot_pool_fixed_3.csv`

**Output:** `predictions_rag_dynamic.csv`, `pool_embeddings.npz` (auto-generated)

Run all cells in order. The notebook will:
1. Load the evaluation set and few-shot pool
2. Embed the full pool using `text-embedding-3-small` — **cached to `pool_embeddings.npz` after first run**; subsequent runs load from cache automatically
3. For each test comment, retrieve the top-7 most semantically similar pool examples via cosine similarity (similarity floor: 0.10)
4. Run async GPT-4.1 inference with dynamically retrieved examples per comment
5. Save predictions and print full evaluation report including embedding cost breakdown

---

### Method 4 — Label-Aware RAG — `FINAL_METHOD4.ipynb`

**Required files:** `stress_test_eval_set.csv`, `few_shot_pool_fixed_3.csv`

**Output:** See Output Files Reference table below

This is the most sophisticated pipeline. Run the workflow functions **in order** in the notebook:

```python
# Step 1 — Split evaluation set into dev and held-out test (run once)
split_eval_set()

# Step 2 — Build label-aware embedding index (run once)
# Also generates curated_hard_labels_cl1.json for optional manual review
build_index()

# [Optional] Step 2b — Manual curation
# Open curated_hard_labels_cl1.json, set "keep": true for high-quality
# hard-label examples, then re-run build_index() to incorporate them

# Step 3 — Classify the dev set
await run_inference_async(split="dev")

# Step 4 — Tune per-label thresholds on dev set with recall floors
best_thresholds = tune_thresholds_on_dev()

# Step 5 — Classify held-out test set using tuned thresholds
best_thresholds = load_dev_thresholds()
await run_inference_async(split="test", thresholds=best_thresholds)

# Step 6 — Final evaluation on held-out test set
evaluate_predictions("test")
```

Key features of Method 4:
- Per-label threshold tuning on dev set (recall floors: `severe_toxic` and `threat` ≥ 0.50; `identity_hate` ≥ 0.60)
- Dedicated severity check pass for borderline `severe_toxic` cases
- Dedicated identity check pass for borderline `identity_hate` cases
- Adaptive self-consistency voting for hard labels only (`severe_toxic`, `threat`, `identity_hate`)
- Skip-gate for high-confidence clean comments (~8–12% bypass GPT calls, reducing cost)
- Post-processing heuristic rules enforcing label hierarchy consistency

## Important Notes

**API keys:** Never commit API keys to the repository. All notebooks use `os.getenv("OPENAI_API_KEY")` — set the key as an environment variable only.

**Cost:** Running all four GPT-4.1 pipelines on the full evaluation set incurs API costs. Method 4 is the most expensive due to multi-pass inference (severity check, identity check, and self-consistency passes for borderline cases). The skip-gate in Method 4 reduces cost by bypassing GPT for high-confidence clean comments.

**Runtime:** Methods 1–3 take approximately 30–60 minutes on the full evaluation set at `ASYNC_CONCURRENCY=8` on a paid OpenAI tier. Method 4 is longer due to `build_index()` (run once) and multi-pass inference.

**Evaluation set design:** The stress-test evaluation set deliberately over-represents rare classes relative to the natural Jigsaw distribution. Macro-F1 scores are not directly comparable to published Jigsaw leaderboard results.

**Claude Haiku 3:** A Claude Haiku 3 pipeline was tested on a 1,000-row subset; results are reported in the Appendix. Full-scale evaluation was not feasible due to free-tier API rate limits restricting concurrency to one request at a time.
